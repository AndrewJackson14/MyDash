# MyDash Audit — 2026-04-20

Quick-pass security + performance audit across the MyDash codebase (React frontend, Supabase migrations, edge functions). Findings are prioritized by severity and blast radius. Each item is actionable as written, with file:line pointers.

**Spot-checked before reporting:**
- `.env` is in `.gitignore` ✅ (the "hardcoded keys in git" critique was a false positive)
- `bunny-storage` has no inline JWT check, but Supabase edge functions enforce `verify_jwt: true` by default unless disabled via `config.toml` (no local config found — assume default is active)
- Migration `004_phase2_schema_expansion.sql` does `ENABLE ROW LEVEL SECURITY` on invoices/payments/subscribers/tickets ✅ — the "RLS not enabled" critique was wrong. The real risk is policy **permissiveness**, which is what's flagged below.

## Critical — address this week

### S-1. Commission tables are open to every authenticated user

[`supabase/migrations/015_commission_system.sql`](supabase/migrations/015_commission_system.sql)

`commission_ledger`, `commission_payouts`, `commission_rates` have RLS enabled but policies are `USING (true)` on all operations. Every logged-in user can read every salesperson's commission history and rates, and can insert arbitrary ledger rows.

**Fix:** scope SELECT to `salesperson_id = auth.uid()` via team_members join, OR admin; scope INSERT/UPDATE/DELETE to admin-only.

### S-2. `team_notes` wide open

[`supabase/migrations/018_spec_new_tables.sql`](supabase/migrations/018_spec_new_tables.sql)

Any authenticated user can read, edit, or delete any other user's DMs, task comments, and MyHelper Q&A. Same `USING (true)` pattern.

**Fix:** SELECT restricted to `from_user = auth.uid() OR to_user = auth.uid()`; UPDATE/DELETE restricted to `from_user = auth.uid()` (so the sender can edit, the recipient can mark read but not alter content).

### S-3. `bunny-storage` path traversal

[`supabase/functions/bunny-storage/index.ts:22-24`](supabase/functions/bunny-storage/index.ts)

`x-path` and `x-filename` headers are concatenated into the BunnyCDN URL with no sanitation. A caller can pass `x-path=../other-pub/` or `x-filename=../../secrets` and read or write outside their pub's folder. JWT auth protects *who* can call the function; it does not protect *what path* they target.

**Fix:** reject any path containing `..`, leading `/` or `\`, or control chars before building the URL; enforce a `pubs/<pub-id>/` prefix derived from the authenticated user's `publications` row, not from client input.

### P-1. Main JS chunk is 529 KB · pdf chunk is 443 KB

[`dist/assets/index-*.js`](dist/assets/), [`dist/assets/pdf-*.js`](dist/assets/)

Initial page load ships ~1 MB of JS (pre-gzip). `pdfjs-dist` appears bundled into the main chunk despite only being used in EditionManager (which has a dynamic import). `tiptap` is ~376 KB loaded on any page that imports EmailTemplates / Mail / StoryEditor.

**Fix:** audit the import graph — find what's defeating the lazy-load of pdfjs-dist (likely a static re-export somewhere). Same investigation for tiptap. Estimated boot-time savings: 500 ms – 1.5 s on mid-tier mobile.

## High — address this month

### S-4. Website-admin tables world-readable

[`supabase/migrations/019_website_admin_tables.sql:78-88`](supabase/migrations/019_website_admin_tables.sql)

`daily_page_views`, `newsletter_subscribers`, `site_errors`, `distribution_points` use `USING (true)` + `WITH CHECK (true)`. Newsletter subscriber emails + addresses are readable by any authenticated user.

**Fix:** restrict `newsletter_subscribers` to publishers/admins. `daily_page_views` read = authenticated is OK (low sensitivity) but INSERT should stay anon-only (already is). `site_errors` admin-only read.

### S-5. Impersonation only gated client-side

[`src/App.jsx:65-74`](src/App.jsx)

`isAdmin` and `impersonating` live in React state; `currentUser = impersonating || realUser` swaps the acting identity, but all DB writes continue to flow through the *real* user's Supabase session. This is actually fine for server-side authz (RLS sees real user), but means impersonation is cosmetic only — "act as user X" doesn't work at the data layer.

**Fix:** document the intent. If it's "admin preview of another role's UI," rename to `viewAs` and leave writes alone. If the goal is truly "admin acts as user X," add audit_log rows per impersonated action and a server-side verification step.

### S-6. `bunny-storage` leaks upstream error text

[`supabase/functions/bunny-storage/index.ts:38-41, 62-66, 85-88`](supabase/functions/bunny-storage/index.ts)

BunnyCDN error bodies — which can contain zone names, path prefixes, occasionally key fragments on 401s — are echoed verbatim in the `detail` field of the response JSON.

**Fix:** `console.error` the full error server-side; return generic `"Upload failed"` / `"Download failed"` to the client.

### S-7. File upload has no allowlist

[`supabase/functions/bunny-storage/index.ts:75-89`](supabase/functions/bunny-storage/index.ts)

Upload accepts any content-type, any filename, any size. A user could push `.exe`, `.html` with inline JS, or an unbounded file and exhaust storage.

**Fix:** allowlist extensions `jpg|jpeg|png|gif|webp|pdf|mp4`; reject filenames with control chars or `..`; cap body size (e.g. 100 MB) via `Content-Length` check.

### P-2. Unbounded `.select()` queries

[`src/components/ChatPanel.jsx:24`](src/components/ChatPanel.jsx) · [`src/components/RoleDashboard.jsx:52`](src/components/RoleDashboard.jsx) · [`src/components/RoleDashboard.jsx:462`](src/components/RoleDashboard.jsx)

`.from("team_notes").select("*")` with no `.limit()` or `.range()`. At 100 k rows (DMs + bot logs accumulate fast), that's several MB per request per user per mount.

**Fix:** `.limit(100).order("created_at", { ascending: false })` on every append-only table fetch.

### P-3. Missing composite indexes

Migrations scan:
- `stories(publication_id, slug)` — every article-detail lookup
- `invoices(status, client_id)` — Billing page's main filter
- `sales(status, rep_id)` — SalesCRM jurisdiction filter
- `team_notes(from_user, to_user, created_at DESC)` — Messaging inbox sort

**Fix:** one new migration with `CREATE INDEX CONCURRENTLY`. Each saves ~150–400 ms on its hot query.

### P-4. DashboardV2 re-renders on every shell tick

[`src/App.jsx:536`](src/App.jsx)

DashboardV2 receives 50+ props. Several are rebuilt inline every render (`commissionGoals || []` creates a new array ref, inline arrow handlers, freshly-computed `jurisdiction`). `React.memo` alone wouldn't help — the prop identities churn.

**Fix:** `useMemo`/`useCallback` the unstable inputs in App.jsx before passing down; then wrap `DashboardV2` in `React.memo`. ~200 ms saved per state change that touches the shell.

## Medium — when convenient

### S-8. `stories.body` sanitized at render, not import

Stored as raw HTML (legacy WP). DOMPurify is applied at render time via `dangerouslySetInnerHTML` in StoryEditor, Mail, EmailTemplates, ProposalSign — all properly wrapped. Risk: any future renderer that forgets the wrap. One unsanitized spot = stored XSS affecting every reader.

**Fix:** sanitize on **save/import**, not just render. OR write an ESLint rule banning raw `dangerouslySetInnerHTML` outside a `<SafeHTML>` wrapper.

### S-9. Edge-function user identity not consistently re-derived

Not every function under `supabase/functions/*` pulls `Authorization` and re-verifies the user via `supabaseClient.auth.getUser()`. Several rely purely on the platform `verify_jwt` gate and then trust client-supplied IDs (`x-author-id`-style patterns). A spoofed header would let a caller attribute a write to someone else.

**Fix:** derive user identity inside every function that attributes a DB row; never trust client-supplied IDs.

### P-5. `localStorage` writes every DashboardV2 render

[`src/pages/DashboardV2.jsx:91-113`](src/pages/DashboardV2.jsx)

`useStreakCounter` writes streak state to localStorage on every `doseWins` / `deadlineAlerts` update. 5–10 ms of sync I/O per render × always-mounted dashboard = cumulative hitch.

**Fix:** batch to once-per-day via a memoized dedupe key.

### P-6. `useSignalFeed` fetches unbounded arrays on mount

[`src/hooks/useSignalFeed.jsx:406-422`](src/hooks/useSignalFeed.jsx)

`daily_page_views` fetch with `// eslint-disable-next-line react-hooks/exhaustive-deps` + `[]` deps. Runs once per consumer mount; with two dashboards open (Publisher + TeamMemberProfile preview) it double-fetches.

**Fix:** module-level promise cache keyed to today's date string.

### P-7. Expensive `filter().sort()` unmemoized in Billing

[`src/pages/Billing.jsx:1416`](src/pages/Billing.jsx) · [`src/pages/Billing.jsx:1426`](src/pages/Billing.jsx)

Client-invoice sorts + payment-history sorts re-run every render inside each client card. ~50–100 ms per card at 1 k+ invoices.

**Fix:** `useMemo` keyed to `[clientInvs, payments]`.

## Low — nice to have

### S-10. DOMPurify using default config

Default allows `svg` and `mathml`. Broader than we need. Central `sanitizeHtml()` helper with a strict allowlist would tighten this and make enforcement via ESLint easier.

### P-8. MediaLibrary images lack intrinsic dimensions

[`src/pages/MediaLibrary.jsx:121, 421, 814, 856`](src/pages/MediaLibrary.jsx)

`loading="lazy"` is set, but `width` / `height` are omitted → Cumulative Layout Shift on every grid row.

**Fix:** pass `width`/`height` from `media_assets` (already captured in the backfill script).

### P-9. `supabase.rpc()` calls without timeout or cache

[`src/hooks/useAppData.jsx:1556, 1976, 1983`](src/hooks/useAppData.jsx)

`recalculate_all_commissions` and similar RPCs touch thousands of rows, called synchronously with no timeout and no user progress indicator.

**Fix:** 5-second timeout + toast if slow; cache results for the visible period.

## Needs judgment before fixing

These change product behavior — flagging, not auto-fixing:

- **S-1 / S-2 / S-4** — ownership model for commissions, notes, subscribers. Per-rep visibility? Admin-only? The policy depends on how you want reps to experience the system.
- **S-3** — path traversal guard needs the "legitimate path schema" defined first.
- **S-5** — impersonation semantics is a product question (preview vs. act-as).
- **P-1** — bundle-split investigation requires build instrumentation to see *why* pdfjs is in main.
