# MyDash Full-Scope Audit — 2026-04-23

Scope: performance bottlenecks, security holes, dead code. Preceding audit
(`AUDIT-2026-04-20.md`) cleared 13/17. This pass closed everything else
plus eight new findings discovered along the way.

---

## Security — fixed in this pass

### S1. Edge functions — open relays + JWT bypass (CRITICAL × 5)
Five edge functions accepted unauthenticated requests. All hardened:

| Function | Was | Now |
|---|---|---|
| `upload-image` | open | JWT + path-allowlist + traversal rejection (v20) |
| `invite-user` | open → account takeover via team_member_id rebind | admin-only JWT (v11) |
| `ai-proxy` | open → Anthropic key drain | authenticated JWT + max_tokens cap (v7) |
| `notify-mention` | open → SES brand-spoofing relay | authenticated JWT (v2) |
| `scheduled-tasks` | open → could trigger arbitrary cron passes | service_role JWT (v12) |
| `contract-email` | open → Gmail-from-staff relay | signer-email lock + 10-min freshness window (v10) |
| `generate-delivery-report` | open → DoS via repeated schedule_id runs | authenticated/service_role JWT (v7) |

CORS also locked to `mydash.media` + dev hosts on all seven (was `*`).

Note: `contract-email` cannot use a real JWT gate — the signer is
unauthenticated. Defense-in-depth was applied: the function ignores the
request `to_email` and uses `proposal_signatures.signer_email`, refuses if
`signed_at` is older than 10 minutes, and forces the subject server-side.

### S2. XSS sinks in LegalNotices
- `LegalNotices.jsx:892` rendered `viewNotice.bodyHtml` raw — sourced from
  the public StellarPress FBN form. Wrapped in `sanitizeHtml()`.
- `LegalNotices.jsx:1091` rendered attachment `href` from the same public
  form with no scheme check. Added `safeAttachmentUrl()` rejecting any
  scheme that isn't http(s).

## Performance — fixed in this pass

### P1. Unbounded loaders (12)
All capped with explicit `.limit()` so the boot payload doesn't drift with
table growth. Caps are 2-5× current row counts:

| Loader | Cap added |
|---|---|
| `proposals` | 2000 |
| `proposal_lines` | 10000 |
| `editions` | 1000 |
| `service_tickets` | 2000 |
| `ticket_comments` | 5000 |
| `legal_notices` | 2000 |
| `legal_notice_issues` | 5000 |
| `creative_jobs` | 2000 |
| `ad_projects` | 2000 |
| `commission_issue_goals` | 2000 |
| `salesperson_pub_assignments` | 1000 |
| `commission_rates` | 1000 |
| `outreach_campaigns` | 500 |
| `outreach_entries` | 5000 |
| `drop_locations`, `drivers`, `routes`, `stops` | 500-5000 |
| `my_priorities` | 500 |

When any of these saturates the cap in production, build a paginated loader
on the consumer screen rather than raising the cap globally.

### P2. Boot bundle bloat
- `IssueDetail` was eagerly imported in `App.jsx`; now `lazyLoad`'d (saves
  ~25KB from the initial chunk for users who never open an issue detail).
- `JSZip` was statically imported by `storyImagesBundle.js` (used only by
  the Story Editor's "Download Originals" button); now dynamically imported.
  The 97KB jszip chunk is now its own lazy split — visible in the build
  output as `jszip.min-*.js`.

### P3. KanbanCol re-render
`EditorialDashboard.jsx` `KanbanCol` was recreating on every parent render
even when its props were unchanged. Wrapped in `memo`. Visible improvement
on workflow boards with many stories.

### P4. IntegrationsPage 1s polling
The OAuth popup callback used a `setInterval(..., 1000)` polling
`localStorage`. Replaced with the `storage` event — fires synchronously
when the popup writes, no idle CPU spend, no battery drain in background
tabs.

## Pending — known but not yet fixed (LOW priority)

- `qb-auth/index.ts:1625` + `gmail-auth/index.ts:1166` — reflected XSS in
  the OAuth error page render. Mitigated by the OAuth flow being a popup
  initiated from authenticated UI; user can't realistically be tricked
  into hitting it with attacker-supplied params. Fix: wrap `error` in
  `JSON.stringify` before string-interpolating.

- `site-errors/index.ts` — could use a publication_id allowlist + a rate
  limit to harden against site-form noise. Currently behind a per-site
  shared secret so the abuse surface is small.
