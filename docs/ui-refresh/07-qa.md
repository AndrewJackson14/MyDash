## Grep audit — residual cleanup surface

The Z proxy from Phase 5 captures the bulk of theme tokens. These
counts are what's still hand-painted with literal values inside
component inline styles. Most are safe (they read as monochrome
under Press but don't theme-flip), but they're the long-tail
cleanup that a per-component pass would fix.

| Pattern | Count | Severity | Note |
|---|---|---|---|
| `color: '#…'` literals in `src/components` | 53 | Low | All ink-tone or status-color leftovers; theme-flips by accident because PRESS_DARK.ink and PRESS_LIGHT.ink read close enough to legacy values. |
| `fontFamily: '…'` non-token | 0 | — | All families flow through `var(--font-*)` or `TYPE.family.*`. |
| `fontSize: <number>` literals | 521 | Medium | High count because the codebase has many one-off pixel sizes (chat bubbles, kanban chips, breadcrumb separators). All render correctly; per-component migration would route them through `TYPE.size.*`. |
| `boxShadow:` occurrences | 32 | Medium-High | Press Room rejects shadows. Concentrated in `IssueProofingTab`, `Lightbox`, `MediaModal`, `MyHelperLauncher`, `StoryEditor`, `ProofAnnotationOverlay`, `TeamMemberPanel`, `EblastComposer`, `FuzzyPicker`, `NotificationPopover`, `GmailNotifPopover`. Most are modal/popover overlays where a shadow is the only depth cue absent a better hairline. |

### Recommended follow-up (post-Phase 7, optional)

A targeted Phase 7B pass on the 11 box-shadow consumers would
remove the last visual remnants of the pre-Press Room glass
aesthetic. Recommend doing it as the components are touched for
unrelated work rather than as a single sweep.

The 521 hardcoded font sizes are mostly inside per-component
typographic scales that already feel proportional. Migration to
`TYPE.size.*` would be a refactor for cleanliness, not for visual
change.

## What "fully done" would look like

1. **A browser session** for items 1, 4, 5, 6 (visual + a11y).
2. **A targeted shadow audit** to drop the 11 box-shadow consumers
